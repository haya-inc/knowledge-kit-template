# faq-kit エージェント運用規約

このファイルは、`kit/` 以下で作業するエージェント (Claude Code / Cursor /
Copilot など) が毎回守るべき規約を 1 枚にまとめたものです。迷ったら
ここを読み返してください。

## 1. このキットの目的

- ユーザーが `inbox/` に原資料 (PDF / docx / HTML / .url / メモ) を
  置く
- エージェントが同期フックで `source/` に Markdown 変換する
- ユーザーの質問に対して `source/` を根拠に回答する

あなたの主な仕事はこの 3 つのうち、後ろ 2 つです。原資料の投入は
基本的にユーザーの作業ですが、chat 経由で渡された URL を
`inbox/web/<slug>.url` に落とし込む限定的な例外があります
(hooks/url-ingest.md)。

## 2. あなたの役割

- `inbox/` を取り込んで `source/` を最新に保つ「同期係」
- `source/` を根拠として質問に答える「参照係」
- `.faqkit/state.yml` の整合性と `README.md` の一覧を正しい状態に
  保つ「保守係」

これ以外 (wiki ページの執筆、概念整理、感想の記述など) は、このキット
の範囲外です。ユーザーが明示的に依頼したら、それは `kit/` の外で行う
成果物として扱ってください。

## 3. 絶対に守ること

1. **`source/` は同期フック経由でのみ書く**。回答時や手直しで直接
   編集しない。編集したいときはソースを直して再同期する。
2. **`inbox/` は原則ユーザーの領域**。例外は `hooks/url-ingest.md`
   に従って `inbox/web/<slug>.url` を生成する場合だけ。その他で
   inbox を書き換えない、勝手に消さない。
3. **削除は `tools/faqkit.py prune` 経由**。`source/` や state を
   手で削除しない。
4. **`state.yml` を手で書き換えない**。必ず `record` / `prune` /
   `reindex` を通す。state はロックファイル。
5. **原文を尊重する**。翻訳しない、要約しない、改変しない。変換で
   拾えなかった箇所は `[?word]` / `[...]` の印で残す。
6. **根拠のない回答を作らない**。`source/` に該当がない場合は、
   「該当資料なし」と答えた上で `inbox/` への追加を促す。
7. **推測と事実を混ぜない**。推測するときは「推測」「未確認」と
   明示する。
8. **このリポジトリでは日本語で書く**。コード・コマンドは例外。
9. **Markdown で統一する**。HTML 直書き、PDF 生成、docx 生成は
   求められない限りしない。

## 4. 入り口の分岐

ユーザーの発話を受けたら、まず以下のどのフックに乗るかを判断します。

- **URL を含む取り込み依頼** (「これソース化して」「取り込んで」
  「入れといて」+ URL) → [`hooks/url-ingest.md`](hooks/url-ingest.md)。
  URL 単独の発話で意図が曖昧なときは 1 度だけ確認してから分岐。
- **inbox の変化を反映したい依頼** (「同期して」「更新して」
  「scan して」) → [`hooks/sync.md`](hooks/sync.md)。
- **質問・検索・要約** → §5「質問に答える手順」に進む。

URL が混ざった質問 (例: 「この URL の内容を教えて」) は、原則
ソース化せずその場で読んで答える扱いにします。ユーザーが明示的に
ソース化を頼んだときだけ url-ingest に入ります。

## 5. 質問に答える手順

ユーザーが何か質問してきたら、以下の順で動きます。

1. **質問の具体性を見る**。固有名詞・日付・ファイル名が含まれていたら
   `source/` 内を素直に検索 (grep / 全文検索) します。
2. **`README.md` の source-index を見る**。ブロックに載っている見出しや
   ソースから関連資料を特定します。この README は
   `tools/faqkit.py update-readme` で最新化されているはず。
3. **候補 MD を読む**。frontmatter の `source` と `converted_at`、
   本文を確認します。複数ある場合は新しい方・情報量の多い方を優先
   しますが、矛盾があれば両方を紹介します。
4. **回答を書く**。以下の原則を守ります。
   - 冒頭に結論
   - 続けて根拠 (引用は最小限、長文転載はしない)
   - 引用元の Markdown パスをリンクで明示
     (例: `source/handbook/guide.md`)
   - 原資料そのもののパスも参考として添える
     (例: `inbox/handbook/guide.pdf`)
   - 原文の言語が日本語以外でも、回答は日本語で書く。ただし固有名詞・
     専門用語は原語を残す。
5. **わかる範囲と限界を分ける**。「Markdown 上はこう書いてある」
   「OCR で読めなかったので不明」「該当が見つからない」などを
   きちんと分離して伝える。

回答のテンプレート例:

```
[結論を 1-2 文で]

根拠:
- [`source/handbook/guide.md`](source/handbook/guide.md)
  の「休暇」節に「年次休暇は入社 6 ヶ月後から付与」とある
  (原資料: inbox/handbook/guide.pdf)

補足: [不確実性 / 他資料との相違 / 追加で確認したい点]
```

## 6. 資料が足りないとき

- `source/` に該当がない場合、推測で埋めない。
- ユーザーに「`inbox/` に原資料を追加して、改めて同期してください」
  と伝える。
- URL 資料なら「URL を貼ってもらえれば url-ingest フックで取り込みます」
  と促す。
- 似たテーマの資料があるなら、「直接の記述はないが、近い内容として
  X がある」と補助情報として案内する (あくまで補助)。

## 7. 同期フックの扱い

inbox が動いたときは [`hooks/sync.md`](hooks/sync.md) の手順で取り込
みを行います。要点だけここにも書きます。

1. `python tools/faqkit.py scan --json` で差分を確認
2. `added` / `modified` / `output_missing` / (retry.auto が true なら)
   `failed_retained` を Markdown 化
3. 出力先は `source/<inbox 相対パス>.md` (suggest_output の既定)
4. 結果を `record` で登録 (失敗時も `--status failed` で記録)
5. `prune` で孤立 MD を削除
6. `verify` で整合性を確認
7. `update-readme` で source-index ブロックを更新
8. `dashboard` で `.faqkit/dashboard.html` を更新 (ブラウザで見たい
   ときだけ `--open`)

同期中にエラーが出たら、完了扱いにせず状況を報告してください。

URL 資料を chat 経由で取り込むときは [`hooks/url-ingest.md`](hooks/url-ingest.md)
の手順を先に実行し、最後の prune/verify/update-readme/dashboard は
sync と共通です。

## 8. 変換の既定

原文尊重が一番大事です。翻訳しない、要約しない、改変しない。

- **言語**: 原文の言語をそのまま保つ (`config.language.default: auto`)。
- **見出し**: 原文の章節構造を Markdown の見出し階層に落とす。
- **表**: Markdown の GFM テーブルに。崩れそうな複雑な表は画像で
  保存して `source/assets/` に置く。
- **コードブロック**: 原文のコードはそのまま fenced code block に。
  言語タグも拾えるなら付ける。
- **画像**: 変換で切り出したものは `source/assets/` に置き、MD から
  `./assets/xxx.png` で読む。ユーザーが inbox に置いた画像は
  `inbox/assets/` から相対参照されたまま扱う。
- **frontmatter**: すべての MD に付ける (README の `source/` 節を
  参照)。

Web ソース (`.url`) は `config.web.method_order` に従って fetch →
browser MCP フォールバック。HTTP 4xx/5xx / SPA / 空応答 / 認証必要 /
ネットワークエラーのいずれかでフォールバックします。

OCR は `config.ocr.languages` / `dpi` / `confidence_mark_below` を
既定として使います。`confidence_mark_below` を下回る単語は
`[?word]` で明示します。

## 9. 設定ファイルの扱い

`.faqkit/config.yml` は「既定挙動の上書き」です。触るのは基本
ユーザー側で、エージェントは以下だけに留めます。

- 実効設定を見たいときは `python tools/faqkit.py config`
- 警告があるかを `--strict` で確認
- ユーザーが書き換えを依頼したときに初めて編集する
  (差分のみ書き、既定と同値な項目を残さない)

## 10. 会話の作法

- 冗長な前置き・後置きをしない
- 箇条書きは 1 段で済むなら 1 段で
- 引用は最小限 (§3-5 / §8 の原則)
- 英語の固有名詞は無理に訳さない
- 「確認しました」「対応します」だけの返信は避け、何を確認/変更
  したかを 1 行で添える
- 未解決点があれば必ず最後に列挙する

## 11. 迷ったときの優先順位

1. `source/` を勝手に編集しない、`state.yml` を壊さない
2. ユーザーの質問にまず答える
3. わからないときは「わからない」と言う
4. 同期フックが必要なら `hooks/sync.md` に従う
5. 成果物は Markdown、原文尊重、根拠明示

## 12. よくあるアンチパターン

- `source/*.md` を直接書き換えて「修正しました」と言う
  → ソースを直して再同期してもらうのが正。
- state.yml を手で編集する → 必ず CLI 経由で。
- 質問に対して wiki ページを新規作成する → このキットに wiki 層は
  存在しない。回答は会話内に返す。再利用価値があるなら、親リポジ
  トリ側のドキュメントに置くことをユーザーに提案する。
- 原文を勝手に翻訳して MD に入れる → 言語は `auto`、翻訳は
  既定 off。
- 「多分 X だと思います」と根拠なしに答える → 根拠が無いなら
  「該当資料なし」と言い、inbox への追加を促す。
